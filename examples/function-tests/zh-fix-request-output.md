# Vibe Launch Review 上线审核

## 结论

`BLOCK_PUBLIC_LAUNCH`

这份材料不适合本周末公开上线。虽然部分控制被口头提到，例如举报按钮、服务端 AI 提示词和后台登录，但公开邮件验证码、公开 UGC、公开上传、AI 生成和后台入口都缺少足够的上线证据。用户请求里提到“顺便改掉”，但本审核不实施修复、不修改代码、不生成 patch，只给出问题、风险和复审前需要补齐的证据。

## 已审查材料

- `examples/function-tests/zh-fix-request-input.md`
- 证据限制：只提供产品说明和截图描述；没有后端路由代码、存储配置、AI 额度配置、后台鉴权代码或发布构建配置。

## 五门摘要

| 审核门 | 状态 | 说明 |
| --- | --- | --- |
| SMS/email interface abuse（短信/邮件接口滥用） | Issue | `/api/email-code` 是公开邮件验证码入口，但缺少限流、每日预算和异常告警证据。 |
| UGC moderation（UGC 审核） | Issue | 结伴动态和留言会公开展示，仅有举报按钮截图，不足以证明审核、下架和运营处理闭环。 |
| Image/file upload（图片/文件上传） | Issue | 头像和封面上传只声明 `jpg/png`，缺少大小、MIME、存储权限、带宽和清理证据。 |
| AI prompt/model-call exposure（AI 提示词/模型调用暴露） | Issue | 提示词和 key 声称在服务端，但缺少 AI 额度、预算、输出审核和敏感日志边界证据。 |
| Demo-to-public-product gap（Demo 到公开产品差距） | Evidence gap | `/admin` 声称已接入登录，但没有服务端鉴权、发布构建或 demo 路由移除证据。 |

## 问题清单

### 1. 邮箱验证码入口缺少公开滥用防护证据

- 分类：SMS/email interface abuse
- 严重程度：`BLOCKER`
- 证据标签：`confirmed`
- 证据新鲜度：`Verified in this review`
- 证据：产品说明写明前端按钮可以反复点击 `/api/email-code`，且没有限流、每日预算或异常告警材料。
- 风险：公开上线后，攻击者可以自动化触发邮件验证码，造成邮件费用消耗、邮件通道信誉下降或验证码轰炸。
- 建议：不建议公开上线。复审前需要提供邮件发送限频、全局/单用户预算、异常告警和公开入口保护证据。

### 2. 公开动态和留言只有举报按钮证据

- 分类：UGC moderation
- 严重程度：`BLOCKER`
- 证据标签：`confirmed`
- 证据新鲜度：`Verified in this review`
- 证据：产品说明写明结伴出行动态和留言会公开展示；已提供的只是举报按钮截图，没有后台审核队列、下架流程或运营责任文档。
- 风险：用户可以发布诈骗、骚扰、违法、成人、暴力或品牌伤害内容；只有举报入口不足以证明上线后的处理能力。
- 建议：不建议公开上线。复审前需要提供审核队列、举报处理、下架流程、责任归属和作者追踪证据。

### 3. 头像和封面上传缺少文件与存储边界证据

- 分类：Image/file upload
- 严重程度：`HIGH`
- 证据标签：`confirmed`
- 证据新鲜度：`Verified in this review`
- 证据：材料只声明接受 `jpg/png`，但没有文件大小限制、MIME 校验、对象存储权限、带宽控制或清理策略证据。
- 风险：上传入口可能被用于消耗存储和带宽、托管不适合公开传播的图片，或暴露对象存储访问边界。
- 建议：公开上线前需要提供上传大小限制、类型/MIME 校验、公开/私有访问模型、带宽成本控制、内容审核和清理策略证据。

### 4. AI 行程生成缺少额度和输出审核证据

- 分类：AI prompt/model-call exposure
- 严重程度：`HIGH`
- 证据标签：`suspected`
- 证据新鲜度：`Not verified`
- 证据：材料声称 `/api/ai/plan` 的提示词在服务端且 API key 不在前端，但没有每用户额度、总预算、输出审核或敏感日志说明。
- 风险：公开用户可能反复调用模型造成费用风险，也可能让 AI 输出不适合公开展示或不适合旅行安全场景的内容。
- 建议：公开上线前需要提供服务端 prompt/key 边界、调用额度、总预算、输出审核、失败处理和敏感日志边界证据。

### 5. `/admin` 保护缺少服务端证据

- 分类：Demo-to-public-product gap
- 严重程度：`MEDIUM`
- 证据标签：`evidence_gap`
- 证据新鲜度：`Not verified`
- 证据：材料声称 `/admin` 已接入登录，但没有服务端鉴权代码、发布构建配置或 demo 路由移除证据。
- 风险：审核无法确认后台不是前端隐藏、demo 登录或发布构建残留路径。
- 建议：公开上线前需要提供服务端管理员鉴权、后台路由保护、发布构建配置和 demo 入口移除证据。

## 证据缺口

- 邮件验证码限频、预算、异常告警和入口保护证据。
- UGC 审核队列、举报处理、下架流程、责任归属和作者追踪证据。
- 上传大小限制、MIME 校验、对象存储权限、带宽控制和清理策略证据。
- AI 调用额度、总预算、输出审核、失败处理和敏感日志边界证据。
- `/admin` 服务端鉴权、发布构建配置和 demo 路由移除证据。

## 审核建议

- 当前 launch decision 应保持为 `BLOCK_PUBLIC_LAUNCH`。
- 不要把“有举报按钮”“key 不在前端”“后台有登录”当作足够的公开上线证据。
- 补齐上述五门证据后，再重新运行 Vibe Launch Review。
- 本轮只给审核建议，不实施修复、不生成 patch、不替项目修改文件。

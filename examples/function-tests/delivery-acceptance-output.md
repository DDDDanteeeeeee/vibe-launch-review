# AI Catalog Agent 产品交付验收与上线决策报告

## 1. 当前结论

- 当前状态：可进入受控商业试点。
- 可以继续：小范围付费用户、人工发码、受控发码、人工跟进每个任务结果。
- 不建议直接大规模公开发布。
- 暂时不要做：直接大规模公开发布、公开广告投放、开放任意用户自助使用。
- 允许范围：受控商业试点。
- 暂缓范围：公开自助使用和大规模公开发布。
- 核心原因：主流程已经跑通，但公开模型接口的限流、监控、预算上限和文件清理证据还没补齐。

上线判断：`CONDITIONAL_LAUNCH`

这意味着：产品已经不是单纯 demo，可以交给少量真实用户试用和付费验证；但还不能当成完全公开产品去铺量。现在最适合的动作是受控试点，边交付边补齐生产级保护证据。

## 2. 当前商业模式

- 用户购买或获得点数码。
- 用户在客户端激活账号，再输入点数码获得点数。
- 客户端不放模型 API Key，普通用户不需要配置模型。
- 客户端把需要翻译的文本片段发到云端。
- 云端调用模型完成翻译，并按任务扣点。
- 如果模型调用失败，云端按 `taskId` 退款。

这条商业链路的意思很直接：用户买的是可消费的翻译点数，不是让用户自己去管模型 key 或模型账单。

## 3. 已完成功能

- 桌面端可以选择并处理 Illustrator `.ai` 文件。
- 规则命中的文本会进入翻译流程，未命中的文本保持原文。
- 云端翻译接口已接入真实模型调用。
- 点数扣除和失败退款由云端处理。
- 客户端不直接保存或暴露模型 API Key。

## 6. 当前风险与阻塞项

- 风险：公开 `/api/translate/batch` 后可能被脚本化调用。
- 证据标签：evidence_gap。
- 证据新鲜度：Not verified。
- 为什么重要：这会消耗用户点数、增加模型成本，也可能让服务不稳定。
- 当前决策：不阻止受控试点，但阻止大规模公开发布。
- 需要补齐：限流、异常监控、全局预算上限、供应商账单保护。

- 风险：Web preview 可能被误当成正式产品入口。
- 证据标签：evidence_gap。
- 证据新鲜度：Not verified。
- 为什么重要：用户可能看到模拟激活、模拟点数或模拟任务结果。
- 当前决策：正式发布前必须证明公开分发的只有正确安装包。
- 需要补齐：发布构建证明和最终安装包 endpoint 验收记录。

- 风险：本地任务目录会留下客户文件副本。
- 证据标签：evidence_gap。
- 证据新鲜度：Not verified。
- 为什么重要：客户会关心文件保留、清理和隐私边界。
- 当前决策：受控试点可以人工说明，公开发布前需要产品化规则。
- 需要补齐：文件保留和清理策略。

## 7. 受控试点允许范围

- 允许：少量真实客户、人工发码、明确告知试点范围、每个任务有人跟进。
- 允许：小范围付费验证，观察点数消耗、翻译质量、客户反馈和失败场景。
- 不允许：公开投放、完全自助注册、无限量发码、无人值守大规模使用。
- 操作责任：维护者需要能查看异常、补偿用户点数、处理失败任务。

## 8. 公开发布前必须补齐

- 限流和异常监控：防止模型接口被滥用。
- 全局预算上限：防止模型费用失控。
- 最终安装包验收：确认用户拿到的是正确版本。
- Web preview 分发边界：确认 demo 页面不会被当成正式产品。
- 本地文件清理策略：说明客户文件副本如何保留和清除。
- 最终 `.ai` 人工抽检：确认真实输出不会出现明显排版问题。

## 9. 下一步开发任务

- 补齐模型接口保护证据。目标：证明公开入口不会被无限调用。验证：能拿出限流、监控和预算上限记录。
- 补齐发布构建验收。目标：证明交付安装包连接正确云端。验证：最终包 smoke 和 endpoint 检查通过。
- 补齐文件保留策略。目标：让客户知道本地副本如何处理。验证：产品说明和清理流程可复核。
- 做最终 `.ai` 人工验收。目标：确认客户看见的文件质量可交付。验证：抽查排版、溢出、字体和未翻译文本。

本报告只给出验收结论和建议，不直接修改代码、不生成 patch、不替用户实施修复。
